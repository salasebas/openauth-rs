# openauth-scim

Server-side SCIM 2.0 support for OpenAuth.

This crate provides the OpenAuth SCIM plugin: provider connection management,
bearer-token authentication, metadata endpoints, SCIM User routes, organization
Group routes, Bulk operations, search, filtering, sorting, pagination,
projection, ETags, and database schema contributions.

OpenAuth uses Better Auth's `@better-auth/scim` package as behavioral guidance,
but this crate is not a line-by-line port. The public API is Rust-first,
server-first, and intentionally omits browser SDKs, dashboard UI, and Better
Auth Infrastructure self-service features.

## Installation

Use the crate directly:

```toml
[dependencies]
openauth-scim = "0.0.5"
```

Or enable SCIM through the public facade crate:

```toml
[dependencies]
openauth = { version = "0.0.5", features = ["scim"] }
```

Register the plugin with OpenAuth:

```rust
use openauth_core::options::OpenAuthOptions;
use openauth_scim::{scim, ScimOptions};

let options = OpenAuthOptions {
    plugins: vec![scim(ScimOptions::default())],
    ..OpenAuthOptions::default()
};
```

SCIM clients call the routes under your auth base URL. For example, if your
auth handler is mounted at `https://app.example.com/api/auth`, the SCIM base URL
is:

```text
https://app.example.com/api/auth/scim/v2
```

Your HTTP integration must allow `GET`, `POST`, `PUT`, `PATCH`, and `DELETE`.

## Provider Connections and Tokens

SCIM requests are authenticated with a bearer token generated by:

```text
POST /scim/generate-token
```

The request requires an authenticated OpenAuth session:

```json
{
  "providerId": "okta",
  "organizationId": "org_123"
}
```

`organizationId` is optional. When it is present, the organization plugin must
be installed and the session user must have one of the allowed organization
roles.

The returned `scimToken` is a base64url token encoded as:

```text
base64(scimToken:providerId[:organizationId])
```

`providerId` must not contain `:`. Organization IDs may contain `:` because the
decoder treats everything after the provider ID as the organization scope.

Management endpoints:

| Method | Path | Purpose |
| --- | --- | --- |
| `POST` | `/scim/generate-token` | Create or rotate a provider token |
| `GET` | `/scim/list-provider-connections` | List providers manageable by the current session |
| `GET` | `/scim/get-provider-connection?providerId=...` | Inspect a provider connection without exposing the token |
| `POST` | `/scim/delete-provider-connection` | Delete a provider and invalidate its token |

## Options

`ScimOptions::default()` matches Better Auth's default token storage behavior:
plain token storage. For production, prefer hashed, encrypted, or
application-owned custom storage.

```rust
use openauth_scim::{ScimOptions, ScimTokenStorage};

let options = ScimOptions {
    token_storage: ScimTokenStorage::Hashed,
    required_role: Some(vec!["owner".to_owned()]),
    ..ScimOptions::default()
};
```

Supported options:

| Field | Default | Behavior |
| --- | --- | --- |
| `default_scim` | empty | Static providers for tests or local provisioning without database rows |
| `required_role` | `admin` plus organization creator role, normally `owner` | Roles allowed to create and manage organization-scoped providers |
| `provider_ownership.enabled` | `false` | Links personal providers to the session user who generated them |
| `token_storage` | `Plain` | `Plain`, `Hashed`, `Encrypted`, custom hash, or custom encrypt/decrypt |
| `before_token_generated` | none | Hook after built-in authorization and before persistence |
| `after_token_generated` | none | Hook after the provider record is persisted |

Default SCIM providers use the plain base token regardless of database storage
mode:

```rust
use openauth_scim::{DefaultScimProvider, ScimOptions};

let options = ScimOptions {
    default_scim: vec![DefaultScimProvider {
        provider_id: "default-okta".to_owned(),
        scim_token: "local-secret".to_owned(),
        organization_id: None,
    }],
    ..ScimOptions::default()
};
```

## SCIM Resource Routes

User routes are available for personal and organization-scoped providers:

| Method | Path | Purpose |
| --- | --- | --- |
| `GET` | `/scim/v2/Users` | List users for the provider scope |
| `POST` | `/scim/v2/Users` | Create a provisioned user and provider account |
| `GET` | `/scim/v2/Users/:userId` | Fetch a user in the provider scope |
| `PUT` | `/scim/v2/Users/:userId` | Replace supported user fields |
| `PATCH` | `/scim/v2/Users/:userId` | Apply SCIM PatchOp operations |
| `DELETE` | `/scim/v2/Users/:userId` | Delete the provisioned user |
| `POST` | `/scim/v2/Users/.search` | Search users with SCIM query parameters in the body |

Group routes are backed by OpenAuth organization teams and therefore require an
organization-scoped SCIM provider:

| Method | Path | Purpose |
| --- | --- | --- |
| `GET` | `/scim/v2/Groups` | List team-backed SCIM groups |
| `POST` | `/scim/v2/Groups` | Create a team-backed group |
| `GET` | `/scim/v2/Groups/:groupId` | Fetch a group |
| `PUT` | `/scim/v2/Groups/:groupId` | Replace group display name and members |
| `PATCH` | `/scim/v2/Groups/:groupId` | Patch group display name or members |
| `DELETE` | `/scim/v2/Groups/:groupId` | Delete the group |
| `POST` | `/scim/v2/Groups/.search` | Search groups |

Additional SCIM 2.0 routes:

| Method | Path | Purpose |
| --- | --- | --- |
| `POST` | `/scim/v2/.search` | Search users and groups together |
| `POST` | `/scim/v2/Bulk` | Execute scoped User and Group operations |
| `GET` | `/scim/v2/Me` | Returns SCIM `501`; provider-scoped tokens are not end-user aliases |
| `GET` | `/scim/v2/ServiceProviderConfig` | Advertise supported SCIM features |
| `GET` | `/scim/v2/Schemas` | List supported SCIM schemas |
| `GET` | `/scim/v2/Schemas/:schemaId` | Fetch a schema |
| `GET` | `/scim/v2/ResourceTypes` | List supported resource types |
| `GET` | `/scim/v2/ResourceTypes/:resourceTypeId` | Fetch a resource type |

## Protocol Behavior

OpenAuth advertises the capabilities it actually implements:

- `patch.supported: true`
- `bulk.supported: true`
- `filter.supported: true`
- `sort.supported: true`
- `etag.supported: true`
- `changePassword.supported: false`

List and search endpoints support `filter`, `sortBy`, `sortOrder`,
`startIndex`, `count`, `attributes`, and `excludedAttributes`. Invalid SCIM
filter syntax returns a SCIM error with `scimType: "invalidFilter"`. Extended
attributes, including Enterprise User extension paths, are supported through the
SCIM profile tables.

Mutable User and Group resources include weak ETags. `PUT`, `PATCH`, `DELETE`,
and Bulk operations honor `If-Match`/operation `version` and return `412` for
stale versions.

SCIM error responses use RFC 7644-compatible bodies and
`application/scim+json` where the route is a SCIM protocol route:

```json
{
  "schemas": ["urn:ietf:params:scim:api:messages:2.0:Error"],
  "status": "400",
  "scimType": "invalidFilter",
  "detail": "Invalid filter expression"
}
```

Management routes are session-authenticated OpenAuth API routes, so their
errors use the regular OpenAuth JSON error shape.

## Database Schema

The SCIM plugin contributes these logical models and database tables:

| Logical model | Database table | Purpose |
| --- | --- | --- |
| `scimProvider` | `scim_providers` | Provider ID, stored SCIM token, optional organization scope, optional owner user |
| `scimUserProfile` | `scim_user_profiles` | Extended SCIM user attributes, external ID, version metadata |
| `scimGroupProfile` | `scim_group_profiles` | Extended SCIM group attributes, external ID, version metadata |

`scimProvider.userId` is always present in the schema as an optional field. It
is populated only when `provider_ownership.enabled` is true. Keeping the field
stable avoids migration churn and preserves compatibility across deployments
that toggle provider ownership.

Run your adapter's migration flow after adding the plugin. The SCIM migration
path is tested with:

- SQLx SQLite
- SQLx Postgres
- SQLx MySQL
- tokio-postgres
- deadpool-postgres

The in-memory adapter is supported for tests and in-memory runtime usage, but it
does not provide durable migrations. Redis and Valkey integrations in this
repository are rate-limit stores, not SCIM identity stores. MongoDB is not a
SCIM backend until OpenAuth has a MongoDB `DbAdapter`.

## Better Auth Parity

| Area | Better Auth `@better-auth/scim` | OpenAuth |
| --- | --- | --- |
| Client/dashboard | Package includes client-facing helpers and Better Auth Infrastructure docs | Not ported into this server crate |
| Users | Supported | Supported |
| Provider management | Supported | Supported |
| Token storage modes | Plain, hashed, encrypted, custom | Plain, hashed, encrypted, custom |
| Organization role checks | Supported | Supported |
| Provider ownership | Optional | Optional, with stable optional schema field |
| Groups | Not in upstream package surface | Supported through organization teams |
| Bulk | Advertised unsupported upstream | Supported |
| Sort and ETags | Advertised unsupported upstream | Supported |
| Enterprise User extension | Not part of upstream metadata surface | Supported |

OpenAuth intentionally keeps the server behavior compatible where practical
while using Rust errors, explicit schema metadata, adapter-backed persistence,
and security-focused token comparison.
